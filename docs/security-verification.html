Security roadmap

Security & Verification Roadmap

This page documents the security model Arqis should move toward after the current Arc Testnet MVP. It is a roadmap, not current production behavior.

Current MVP

Prototype access model

The current static MVP uses a browser client and Arc Testnet wallet connection to validate invoice creation, payer inbox, checkout, and seller activity. It is suitable for limited testnet review, not broad public production use.

Known limitation

Client-side trust is limited

Payment status and invoice updates should not rely on the browser as the final source of truth. Future versions should verify payments server-side before marking invoices as paid.

Planned access model

Access matrix to design before tightening RLS

Register Arqis name

A connected wallet should only register or update its own name binding.

Read user lookup

Name and wallet lookup may remain limited and purpose-specific so invoices can be addressed without exposing unnecessary profile data.

Create invoice

Only the seller identity tied to the connected wallet should create invoices from that Arqis name.

Read invoices

Only invoice participants should read sent or received invoice rows, except for narrowly scoped public checkout records if a future external checkout is added.

Update invoice status

Final payment status should be updated by a trusted backend or Edge Function after verification, not directly by arbitrary client updates.

Payment verification

Backend or Edge Function

A future verification service should receive an invoice ID and transaction hash, then verify chain, token contract, recipient, amount, confirmation status, expiry, and duplicate-use protection before updating the invoice.

Status lifecycle

Planned invoice states

unpaid, pending, paid, failed, expired, and cancelled. The current MVP should not overclaim these states until the database and verification flow support them.

Recommendation: Do not tighten Supabase RLS or migrate payment verification until the access matrix, auth/wallet binding approach, and test plan are agreed. This avoids breaking the current prototype while making the security path explicit.
