Future vision

Future Roadmap

These are future product directions. They should not be interpreted as features available in the current testnet prototype. Multi-asset routing, bridge/swap into USDC, and Dust-to-USDC cleanup will be considered after the core USDC invoice flow is validated.

Future pillar

Universal Balance Pay

Allow payers to complete a USDC invoice using value from multiple assets and chains. Arqis can propose a route that swaps or bridges selected assets into USDC, while the seller receives clean USDC settlement on Arc.

Example

Enough value, wrong assets

Invoice: 1,000 USDC. Wallet: 500 USDC + 250 BNB + 100 ETH + 300 BTC value. Arqis suggests how to combine selected assets into one USDC payment.

How it should behave

Payer-controlled routing

Detect available balances

Show USDC and supported assets across chains, with approximate USD value.

Suggest a payment route

Recommend which assets can cover the invoice, including swap, bridge, fee, slippage, and estimated settlement time.

Require explicit confirmation

Never auto-spend, auto-swap, or auto-bridge. The payer must review and approve the route.

Settle cleanly on Arc

The seller sees a simple result: invoice paid, USDC received on Arc, transaction hash available.

Future utility

Dust-to-USDC

When wallet assets are worth less than a threshold, for example under $5, Arqis can ask whether the payer wants to hide small balances or convert eligible balances into USDC.

Example prompt

Small balance cleanup

“You have 7 small balances under $5. Hide them, or convert eligible balances to USDC?”

Important: Dust conversion only makes sense when fees and slippage are lower than the value recovered. Arqis should warn users when conversion is not worth it.
Security note: RLS tightening, backend payment verification, and richer invoice status lifecycle are tracked separately in the Security & Verification Roadmap.

Other future ideas

  • Split payment from multiple wallets
  • Quote lock window before confirming route
  • Payment fallback if a bridge/swap route fails
  • Seller settings for simple/flexible/advanced payment acceptance
  • API + webhooks for route quoted, pending bridge, settled, or failed states

What stays simple for MVP

The current MVP should stay focused on the USDC invoice flow on Arc Testnet: Create Invoice, Pay Invoice, Seller Console, and Circle Faucet / Setup. These future ideas should remain documented as vision, not added to the main UI yet.
